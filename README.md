# CS230-Portfolio

Overview:
This artifact is a software design document that I created for The Gaming Room and their game, Draw It or Lose It. The client wanted to expand their existing Android game into a web-based application that could support multiple users, teams, and operating systems. The goal was to design a scalable solution that would allow players to access the game across different devices while maintaining consistent gameplay and performance.

What I Did Well:
One area I believe I did particularly well was analyzing the different operating platforms and recommending the best solution for the client's needs. I compared Linux, Windows, macOS, and mobile platforms and explained the advantages and disadvantages of each. This helped me better understand how software requirements can influence architectural decisions.

How the Design Document Helped Development:
Working through the design document helped me organize my thoughts before development. Instead of jumping directly into coding, I was able to identify requirements, constraints, system architecture, and user needs first. This process made it easier to understand how the different parts of the application would work together and helped create a stronger foundation for future development.

What I Would Revise:
If I could revise one part of this project, I would expand the system architecture section with additional diagrams and more detailed explanations of how the distributed components communicate. While the design document provides a solid overview, additional visual representations would make the design easier for both developers and stakeholders to understand.

Interpreting User Needs:
I interpreted the user's needs by focusing on the client's goal of expanding Draw It or Lose It beyond Android devices. The design recommendations emphasized scalability, cross-platform compatibility, performance, and security. Understanding user needs is important because software should solve real problems and provide a positive experience for the people who use it.

Software Design Approach:
My approach to software design involved gathering requirements, identifying constraints, evaluating operating platforms, and recommending solutions that best met the client's objectives. In future projects, I would continue using a structured design process that includes requirement analysis, system modeling, architectural evaluation, and security planning before development begins. This approach helps reduce errors and creates a clearer roadmap for building software applications.
